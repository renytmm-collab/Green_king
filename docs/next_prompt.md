# 下一阶段可直接复制的 Prompt

你正在 `D:\code\Green_king` 开发一个独立原创的局域网双人浏览器游戏。请**只**搭建“最小局域网通信原型”，不要实现任何正式游戏玩法。

目标：

1. 一台 Windows 电脑能启动本地服务端；
2. 浏览器 A 可以创建房间；
3. 浏览器 B 可通过同一 Wi-Fi/LAN 的主机 IP 加入房间；
4. 服务端为两人分配 `player1` / `player2`；
5. 两个浏览器之间实时同步鼠标点击位置：任一方点击页面内画布，对方画布显示该点击位置和玩家颜色；
6. 服务端显示或记录连接、建房、加入、点击事件的简短无敏感日志。

推荐技术栈：Node.js + Express + `ws` + 原生 HTML/CSS/Canvas。使用服务端权威房间状态；客户端只发送点击意图与坐标。可以使用 `npm init`、安装最少依赖并创建运行文件，因为本阶段明确授权搭建原型。

实现约束：

- 只创建一个房间或最小房间码即可；房满时给清晰提示。
- 页面提供“创建房间”“输入房间码加入”“连接状态”“我的身份”UI。
- Canvas 使用固定逻辑尺寸；发送归一化坐标（0–1），接收方按自身画布尺寸渲染，避免窗口大小不同造成偏移。
- WebSocket 协议至少包含 `create_room`、`join_room`、`welcome`、`pointer_click`、`opponent_pointer`、`error`。
- 服务端验证：房间存在、未满、玩家身份、坐标为 0–1 的有限数值；不要信任客户端传来的 `playerId`。
- 监听局域网可访问的地址，并在 README 写明如何查看主机 IPv4、让 Windows 防火墙允许端口、以及公共 Wi-Fi 可能启用客户端隔离。
- 提供 `README.md`：安装、启动、主机/另一台电脑的访问步骤、两浏览器手测清单、已知限制。
- 提供适合 Windows 的启动方式（例如 npm script 或 `.cmd`），但不要自动修改防火墙。
- 在实现后做最小验证；如当前环境不能进行真实双机测试，要诚实写明，并给出用户的双机测试步骤。

明确不要实现：金矿、兵营、防御塔、士兵、战斗、建筑升级、胜负、地图玩法、寻路、AI、账号、数据库、公网部署、聊天、音效、美术资产或完整游戏 UI。

开始前先检查当前目录已有文件和 `docs/` 中的规划；随后给出一个简短实施计划并执行。不要下载或复制《The Green Kingdom》的任何原文件、素材、代码或关卡。
