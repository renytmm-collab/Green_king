# 后续开发路线图与额度控制

## 对话拆分原则

每次只交付一个可运行、可验收的小阶段；先阅读现有 README 与相关 1–3 个文件，再改动最小范围。一次对话不同时重写网络、模拟、渲染和美术。轻量模型适合样板、文档、局部 CSS/测试；涉及权威状态、同步协议、战斗逻辑、跨模块排错时使用 Sol 或同等级模型。

| 阶段 | 主要任务与验收 | 难点/风险 | 模型 / 相对消耗 |
| --- | --- | --- | --- |
| 1. LAN 连通验证 | 服务端启动；同机两个浏览器连入；点击坐标互见 | IP/端口/防火墙 | 轻量→Sol 复核 / 低 |
| 2. 房间 | 创建、加入、P1/P2、ready、满员拒绝 | 断线与重复加入 | Sol / 中 |
| 3. 地图和建筑点 | Canvas 渲染一张对称原创地图；点 ID 稳定 | 坐标与状态分离 | 轻量 / 低 |
| 4. 建造与金币 | 服务端校验建造、扣费、矿山产金 | 双重扣费、状态版本 | Sol / 中 |
| 5. 兵营和目标 | 拖拽/点击目标、合法连线、取消路线 | 输入预览与服务端校验 | Sol / 中 |
| 6. 士兵移动战斗 | 定时生成、路径移动、伤害、死亡 | tick 一致性、单位上限 | Sol / 高 |
| 7. 防御塔 | 范围选敌、冷却、投射/伤害 | 目标选择与性能 | Sol / 中 |
| 8. 建筑升级 | 小型原创升级表、成本、UI | 平衡与状态迁移 | 轻量+Sol 审查 / 中 |
| 9. 胜负重开 | 城堡毁坏、结算、双方确认重开 | 房间状态机 | Sol / 中 |
| 10. 双机测试修复 | 两台 Windows、刷新/断线/慢网测试 | 环境差异 | Sol / 中到高 |
| 11. README/脚本 | 明确启动、查 IP、防火墙、加入方式 | 安装步骤准确 | 轻量 / 低 |

## 每阶段验收标准

1. 必须有可重复操作的手工检查步骤与预期结果。
2. 修改后先做最小范围验证；不要因“顺便重构”扩大 diff。
3. 所有判定以服务端日志/状态为准，客户端画面只作呈现。
4. 每完成一阶段，在 README 追加已知限制；发现的问题写入一个简短清单，而不是口头遗忘。

## 周额度的相对规划

**第一周**：阶段 1–5。目标是双机建房、身份分配、坐标同步、静态地图、服务端验证的金币和路线；尚无完整战斗也可验收。

**第二周**：阶段 6–11。完成士兵/塔/城堡闭环、重开、真实双机修复、启动文档。若阶段 6 同步问题未稳定，应延后升级和任何美术。

这不是官方 Plus 固定 token 上限估计；它只表达相对开发量。最大消耗通常来自反复追踪同步分叉、一次性修改多模块、让模型读取大日志/整个依赖目录，以及未经范围约束的美术和“完整复刻”要求。

## 节省 token 的具体方法

- 每次 prompt 写清“只改哪些文件、不要重构、验收命令/手测步骤”。
- 保持一个简短 `README` 和协议/状态机文档，避免每次重复解释架构。
- 让模型先 `rg` 精确查符号，日志仅按关键词取最近 50 条；不读取 `node_modules`、锁文件或大缓存。
- 先固定地图数据、消息 schema 和 tick 规则，再做 UI；避免返工。
- 为每个 bug 给复现步骤、预期/实际结果、两端状态版本，而非粘贴整段控制台日志。
- 大改前拆成“设计确认 → 单模块实现 → 双机验证”三次小对话。
