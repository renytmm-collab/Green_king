# 《绿色王国之战》局域网双人版技术方案

## 目标与边界

制作独立原创、两台 Windows 电脑同 Wi-Fi/LAN 的 1v1 浏览器游戏。一台主机运行本地服务，另一台通过 `http://主机IP:端口` 加入；不使用公网、账号、数据库或原版资产。原版是 PvE 关卡制，局域网版是公平对称 PvP。

## PvP 改造

- 左右镜像地图、相同的城堡/建筑点/初始金币/规则；随机种子仅用于装饰，不能造成资源不对称。
- 原版 AI 的“建造、设路线、取消路线、升级、攻守权衡”全部变为玩家二的鼠标操作。
- 原版关卡压力、敌军预置优势和多王国叙事不带入 PvP；首版仅两个阵营。
- 预防失衡：起始资源相同；同类地块到中央距离相同；塔限制在固定点；兵营总人口上限和单路线兵数上限；先后手同时生效（服务端 tick 统一结算）。

## 鼠标交互决策

| 方案 | 与原版接近度 | 网络/实现 | 结论 |
| --- | --- | --- | --- |
| A 点击兵营→点击目标 | 中等 | 最简单，发送两个 ID | 可作无拖拽备选/无障碍模式 |
| B 从兵营拖拽到目标 | 高 | 发送起点、终点、可选路径点；服务端校验 | **推荐 MVP**：保留原作的“画进攻路线”手感，但首版限制为相邻建筑点的预定义连线 |
| C 自由绘制路线 | 最高 | 路径简化、障碍检测、碰撞、同步均复杂 | 不做首版；原版确有手绘路线证据，但复刻版不必立刻复制该复杂度 |

推荐交互：单击空建筑点弹出建造菜单；选择建筑确认。按住己方兵营，拖到**道路图中直接相连**的敌方建筑即创建/替换路线；再次选兵营可取消路线。客户端只提交意图，不决定是否合法。

## 技术栈比较与选择

| 方案 | 优点 | 代价 | 判断 |
| --- | --- | --- | --- |
| Node.js + Express + `ws` + HTML Canvas | 前后端均 JavaScript；状态对象自然序列化；Windows 启动与浏览器调试直接；依赖少 | 需一套 Node 环境 | **推荐** |
| Python + FastAPI/Flask + WebSocket + Canvas | 服务端逻辑书写清楚 | 前端仍是 JS；依赖/启动与异步 WebSocket 组合更多 | 可行，但不如全 JS 节省上下文 |
| Phaser | 场景、输入、精灵工具更完整 | 添加框架 API、构建配置和依赖；本项目固定点 2D 不需要 | MVP 不采用；若日后动画/UI 扩大再评估 |

推荐：Node.js（LTS）+ Express（静态页）+ `ws`（WebSocket）+ 原生 HTML/CSS/Canvas。只需一个服务进程、一个浏览器端页面和共享 JS 常量；无构建链、无数据库。此处仅是未来实施建议，本次没有安装或搭建任何环境。

## 服务端权威

服务端是唯一真相来源，维护固定时间步（例如 20 Hz）：房间、玩家身份、金币、建筑归属/血量/建造、升级、路线、士兵生成与位置、攻击/伤害、冷却、人口、胜负与重开。

客户端负责渲染、鼠标收集、菜单、临时拖拽预览和接收状态。客户端绝不能自行扣钱、结算伤害或宣布胜利：两台独立浮点/帧率/网络到达顺序会产生分叉，也会让篡改浏览器的玩家伪造金币或单位。

## 房间与消息

流程：主机启动服务 → 浏览器 A 创建房间 → 服务端分配 `player1` 和短房间码 → 浏览器 B 输入主机 LAN IP 与房间码加入 → 分配 `player2` → 双方 `ready` → 服务端开始。断线进入暂停/胜负判定前的重连窗口；MVP 可先实现“任一方断线即结束”。

消息最小集合：

- client → server：`create_room`、`join_room`、`ready`、`build`、`set_route`、`cancel_route`、`restart_request`、`pointer`（仅下一阶段原型）。
- server → client：`welcome`、`room_state`、`match_start`、`state_snapshot`、`action_rejected`、`match_end`、`opponent_pointer`。

每条操作含 `roomId`、递增 `clientActionId` 和所见 `stateVersion`；服务端验证身份、回合状态、所有权、金币、冷却、地块和路线后只执行一次并返回新状态版本。

## 同步策略

首版采用“**操作上行 + 服务端完整状态快照下行**”：操作立即发送，服务端以 20 Hz 模拟并以约 10 Hz 广播精简 JSON 快照；低频 UI 可随快照更新。局域网下无需帧同步、客户端预测、回滚或复杂插值。士兵渲染若有视觉抖动，仅根据相邻快照做线性插值，不能改变真实位置。延迟或重复包由 action ID 和服务器验证处理。

## MVP 范围

**必须有**：一张原创左右对称地图；两名玩家；两座城堡；固定建筑点；金币；矿山、塔、兵营；一种士兵；鼠标设定目标；道路图上的移动和战斗；城堡毁坏；创建/加入房间；胜负/重开界面。

**简化**：几何图形和原创文字；固定点；仅相邻连线；一种兵；无自由路径；最多一条兵营路线；极少或无升级树；单房间；断线即结束。

**不做**：原版 15 关与原版美术/音效；多王国与 AI；成就/评分；公网匹配；账户/数据库/排行榜/聊天/移动端；复杂寻路；预测、回滚、反作弊系统。

## 风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 原版细节证据不足 | 以 `original_game_research.md` 的证据级别为准，原创规则另行记录 |
| 自由路线带来寻路爆炸 | MVP 用固定道路图与服务器端路径表 |
| 不同步/重复点击 | 权威服务端、版本号、action ID、原子校验 |
| PvE 数值不适合 PvP | 使用原创数值表，镜像地图、短局内测、记录胜率 |
| Windows 防火墙或只监听 loopback | 显式监听 LAN 接口/`0.0.0.0`，首次启动提示防火墙规则；测试两台真实设备 |
| 公共 Wi-Fi 客户端隔离/IP 变化 | 说明需非隔离局域网，显示本机 IP，刷新后允许重新加入 |
| 范围膨胀 | 每次对话仅完成路线图的一个可验收阶段 |
