## 三类体验问题修复（用户反馈的 3 个痛点）

### ① 兵营路线可重叠 + 吸附并入共享通道
- 需求：两个兵营路线应可重叠；不必每次重画整条；拖到已有己方路线的任意点（吸附最近点）即并入，继承原目标，真正共用通道。
- 实现（前端 `public/app.js`）：
  - `pointermove` 新增 `computeDragSnap`：拖拽时若光标进入某个己方其他兵营路线的 `ROUTE_SNAP_DISTANCE`(0.045) 内，记录最近点 `drag.snap`（含 routeId/index/point），并把 `drag.previewPoint` 设为该点。
  - `finishDrag`：`current.snap` 存在时，合并路径 = `[b2 起点, ...原路线 points.slice(snapIndex+1)]`，`targetBuildingIds` 继承原路线，发送普通 `create_route`。
  - 渲染：`render` 中 `if (drag?.snap) drawSnapMarker(drag.snap.point)` 画吸附标记。
  - 服务端 `createRoute` 无需改动（烘焙后的合并路径端点仍满足"起点兵营/终点敌建筑"，orthogonalize 后合法；重叠段坐标相同=同一通道）。
- 关键决策：采用"烘焙合并路径"而非运行时引用原路线——逻辑结果与用户描述的"新段+原尾段共用通道"一致，且原路线被取消时不影响本条，实现更稳。

### ② 对局中可退出返回大厅（解散房间）

## ⑤ 还原"抢点"博弈点（点击即预留格子）

- 用户调研需求：原版可在格子变对方色"之前"点它→建造栏出现；即使随后该格翻成对方色，仍可在其上建自己的建筑。现版本不支持（建造时服务端按实时 `cell.territory` 判定，无预留机制）。
- 实现服务端预留（`server.js`）：
  - 房间新增 `reservations: Map`（cellIndex→owner）。`reserveCell(socket,row,col)`：要求 `cell.territory===playerId` 且草/无建筑/无障碍，first-come（已预留则 no-op）；`cancelReservation(socket,row,col)`：仅释放自己预留的格。
  - `build()`：新增 `reservedByMe`/`reservedByOther`；`reservedByOther` 直接拒绝"该格子已被对方预留"；`territory!==playerId && !reservedByMe` 才拒绝"your own green land"。建造成功后 `room.reservations.delete(index)` 消费预留。
  - `destroyBuilding` 清掉该格预留；`leaveRoom` 清掉断开玩家的预留；`resetRoom` 清空；`publicState` 暴露 `reservations:[{row,column,owner}]`；`handleMessage` 注册 `reserve_cell`/`cancel_reservation`；导出两函数。
- 前端接入（`public/app.js`）：`showBuildMenu` 发送 `reserve_cell`；`closeMenus` 用 priorCell 发送 `cancel_reservation`（关闭/切格/建造后释放）；`pointerdown` 中若 `state.reservations` 显示该格被对手预留则提示"该格子已被对方预留"且不弹建造栏；重复点击已预留的同一格 `return` 保住预留；默认 state 增加 `reservations:[]`。
- 视觉（`public/renderer.js`）：`drawReservations(state)` 在已预留格画房主色虚线框；`drawMap` 末尾调用。
- 验证：`npm test` 80/80（新增 8 项：翻面后仍可建/对手不能建预留格/取消后对手可建/先到先得/仅自己可预留/destroy释放/reset清空/publicState暴露）。`node --check` server.js 与 .mjs 前端文件全过；`node server.js` 干净启动 :3000 返回 200。

## ⑥ 路线合并与升级菜单按钮修复（用户截图反馈）

- 问题 1：路线合并没有实现。原因：`ROUTE_SNAP_DISTANCE=0.045` 太小，且 `computeDragSnap` 只检测已有路线上的离散点，不检测线段，导致正常拖拽很难触发吸附。
- 修复（`public/app.js`）：
  - 将 `ROUTE_SNAP_DISTANCE` 从 0.045 提升到 0.12（约 1.5 格），更容易触发。
  - 新增 `projectPointToSegment`，`computeDragSnap` 改为检测光标到已有路线每段线段的距离；吸附点取线段上的最近投影点。

- 问题 2/3：防御塔和城堡的升级菜单错误显示「取消路线」「拆除建筑」。原因：`style.css` 中 `.route-btn { display: flex; }` 优先级覆盖了 HTML `hidden` 属性的默认 `display:none`。
- 修复（`public/style.css`）：新增 `.route-btn[hidden] { display: none; }`，防御塔/城堡菜单里的不相关按钮正确隐藏。

## ⑦ 路线去回头/去回环（用户截图反馈）

- 问题：玩家拖拽路线时出现分叉、回头（如先右再左），导致路线不自然。
- 实现前后端统一的正交路径去回环：
  - `server.js`：新增 `orthogonalSegmentIntersection` 与 `pruneRouteBacktracking`；`orthogonalizeRouteSamples` 返回前调用 prune。遍历正交点序列，用栈维护无交点路径；新线段若与非相邻旧线段相交（含共线反向重叠）则在交点处截断，并丢弃回环之后的部分；同向共线重叠视为共享通道，不截断。
  - `public/app.js`：`orthogonalizePreview` 返回前调用同样的 `pruneRouteBacktracking`，拖拽 preview 也不会显示回头分叉。
- 验证：`npm test` 81/81 全绿（新增 1 项：画右→左→上的回环路径被自动截断为不回头路径）。`node --check` server.js/app.js 全过。

## ⑨ 兵营菜单「取消路线」按钮保持可见（用户反馈）

- 反馈：用户见兵营升级菜单未显示「取消路线」按钮，误以为该功能被删除。
- 原因：此前逻辑只在兵营**已有路线**时显示按钮（`upgradeCancelRoute.hidden = !(building.type === 'barracks' && existingRoute)`）。没有路线的兵营打开菜单时按钮完全隐藏，造成误解。
- 修复（`public/app.js` + `public/style.css`）：
  - `showUpgradeMenu` 改为对兵营**永远显示**按钮：`upgradeCancelRoute.hidden = building.type !== 'barracks'`。
  - 无路线时置灰禁用：`upgradeCancelRoute.disabled = !existingRoute`，标签改为「无路线」；有路线时标签恢复「取消路线」并可点击。
  - CSS 新增 `.route-btn:disabled { opacity: 0.45; cursor: not-allowed; }` 与 `:hover:not(:disabled)`，确保禁用态视觉明确。
- 验证：`node --check public/app.js` 通过；`npm test` 83/83 全绿。
